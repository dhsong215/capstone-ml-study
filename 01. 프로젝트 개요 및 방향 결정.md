# 01. 프로젝트 개요 및 방향 결정 과정

## 1.1 프로젝트 기본 정보

- **프로젝트명**: 바이럴 예측 모델 (식품 키워드 검색 트렌드 기반)
- **팀 규모**: 5명
- **총 예산**: 100만원

---

## 1.2 초기 구상과 수정 과정

### 최초 구상

> 1. 특정 키워드와 피처를 입력하면 해당 상품/키워드의 검색량 피크와 피크 도달 예상 날짜 예측
> 2. 현재 상위 검색어들과 피처들을 입력하면 앞으로 상승 가능성 높은 키워드들 예측

### 문제점 분석 및 수정

#### ①번 구상: "피크 값과 피크 날짜 예측"

**문제**: ML에서 **회귀(Regression)** 문제에 해당하나, 네이버 DataLab 데이터가 0\~100 정규화 값이라 "피크 값" 예측이 무의미함 (피크는 항상 100). 절대 검색량을 모르므로 "하루 5만 번 검색될 것" 같은 예측은 원천 불가.

**"피크 시점 예측"** 은 가능하나, 기존 시계열 모델(Prophet, ARIMA, LSTM)을 그냥 적용하는 수준이라 캡스톤 차별성 부족. 또한 이 모델들은 "이미 상승 중인 트렌드의 미래"를 예측하는 것이지, 아직 안 뜬 키워드가 "언제 뜰지"를 예측하는 게 아님.

#### ②번 구상: "상승 가능성 높은 키워드 예측"

**판단**: 이것이 프로젝트의 핵심이 되어야 함. **분류(Classification)** 문제에 가까우며 학술적으로도 더 가치 있음.

### 최종 방향 결정

**1단계 (메인)**: 식품 키워드의 바이럴 강도(virality_score)와 피크 시점(peak_timing)을 예측하는 **회귀 모델** + 임계값 기반 분류 변환

- virality_score: "얼마나 급등하는가" (Z-score 기반 연속값)
- peak_timing: "향후 30일 중 몇째 날에 피크가 오는가" (1\~30, argmax 기반)
- 두 타겟은 동일한 피처·모델 구조를 사용하고 타겟만 교체하여 학습

**2단계 (보너스, 시간 여유 시)**: 특정 키워드의 향후 30일 검색 트렌드 곡선 자체를 예측하는 **시계열 예측 모델**

2단계가 1단계를 포함하는 것처럼 보이나, 시계열 예측은 매일의 값을 맞혀야 해서 오차가 누적됨(1일 후 ±3 → 30일 후 ±25). 1단계의 peak_timing은 "전체 결론(몇째 날인가)"만 맞히면 되어 훨씬 안정적. 따라서 1단계를 확실히 완성한 후 2단계를 추가하는 것이 안전.

## 1.3 "바이럴"의 정의 — 학술적 근거

### 기존 연구에서의 정의

| 방법 | 설명 | 출처 |
|------|------|------|
| **Z-score 기반** | 평소 평균 대비 표준편차의 N배 이상 급등 시 바이럴 | Facebook 바이럴 탐지 연구 (2022) |
| **이동평균 대비 급등률** | 7일 이동평균 대비 200\~300% 이상 급등 | 일반적 업계 기준 |
| **Kleinberg 버스트 탐지** | 평상/버스트 두 상태의 비용함수로 자동 탐지 | Kleinberg (2002), Cornell |
| **리트윗-팔로워 비율** | retweet/follower > 2.16 | arXiv:2303.06120 (2023) |

### 본 프로젝트 정의

**연속형 타겟 (회귀):**

```
virality_score = (향후 30일 내 최대 검색 트렌드 - 과거 90일 이동평균) / (과거 90일 표준편차 + ε)
# ε = 1e-6 (epsilon smoothing, 0 나누기 방지)
```

**이산형 타겟 (분류 변환):**

- `virality_score > 2.0` → 바이럴 (1)
- `virality_score ≤ 2.0` → 비바이럴 (0)

Z-score > 2를 기본으로 하되, Z-score > 3으로도 실험하여 **임계값에 따른 모델 성능 변화**를 분석 포인트로 활용.

### 기준을 연속 수치로 전환한 이유

| 관점 | 이진 분류 (0/1) | 연속 수치 (virality_score) |
|------|-----------------|---------------------------|
| 클래스 불균형 | 5\~10% vs 90\~95% → 심각한 불균형 | **불균형 개념 자체가 없음** |
| 정보량 | "바이럴이다" 또는 "아니다" | "얼마나 바이럴인가" (6.2 vs 2.1 구분 가능) |
| 학습 기여 | 비바이럴 라벨 전부 동일한 0 | 0.3짜리와 -0.5짜리가 서로 다른 정보 제공 |
| 단점 | 명확한 임계값 필요 | 분포가 극단적으로 치우침 (log 변환으로 완화) |

## 1.4 도메인 선정

### 결정: 식품 > 과자/베이커리

네이버 데이터랩 쇼핑인사이트의 카테고리 체계 활용:
- 1분류: 식품
- 2분류: 과자/베이커리
- 3분류: 하위 세부 카테고리 존재

### 선정 이유

- 식품은 계절성이 뚜렷하여 모델이 학습할 패턴이 풍부
- 과자/베이커리는 바이럴 사례가 빈번 (약과, 탕후루, 크로플, 두바이 초콜릿 등)
- 네이버 쇼핑인사이트 Top 500으로 키워드 풀 자동 확보 가능

## 1.5 추론(서비스) 시나리오

사용자가 키워드 하나만 입력하면 전체가 자동으로 동작:

```
사용자 입력: "약과"
         ↓
[자동 수집 파이프라인]
  ├─ 네이버 DataLab API → 최근 60일 검색 트렌드 자동 조회
  ├─ 썸트렌드 무료 티어 → 최근 블로그/뉴스 언급량·감성 자동 조회
  └─ 날짜 정보 → 요일, 월, 공휴일 자동 계산
         ↓
[자동 피처 계산]
  velocity, acceleration, MA, positive_ratio 등 38개 피처 자동 생성
  (연관어 벡터는 TF-IDF+PCA로 사전 계산, 경제지표는 자동 수집)
         ↓
[모델 추론]
  ├─ XGBoost: 피처 38개 → virality_score 4.2, peak_timing 4일
  └─ LSTM: 60일 시계열 + 연관어 벡터 → virality_score 3.8, peak_timing 5일
         ↓
사용자 출력: "약과 — 상승 강도 4.2, 예상 피크 4일 후, 바이럴 가능성 높음 (즉시 대응 필요)"
```

모든 피처는 API 자동 호출 + 자동 계산으로 채워지며, 사용자가 20개 값을 수동 입력할 필요 없음.

## 1.6 주요 초보자 함정 (학술 연구 기반 경고)

| 함정 | 설명 | 대응 |
|------|------|------|
| **생존자 편향** | 바이럴된 키워드만 모아 학습하면 "뭐가 안 뜨는지" 학습 불가 | 비바이럴 키워드를 같은 비중으로 포함 |
| **클래스 불균형** | 바이럴 5% 미만, "전부 아니오"만 답해도 95% 정확도 | F1-score, ROC-AUC 사용. 회귀 접근으로 근본 해결 |
| **데이터 누수 (Data Leakage)** | 시계열에서 미래 데이터가 학습에 섞이면 성능 뻥튀기 | 시간 기준 train/test 분할, k-fold 금지 |
