## 5.1 타겟 변수 정의

본 프로젝트는 두 가지 타겟을 예측한다:

| 타겟 | 유형 | 의미 | 활용 |
|------|------|------|------|
| **virality_score** | 연속값 (회귀) | 향후 30일 내 얼마나 급등하는가 | 메인 타겟 |
| **peak_timing** | 연속값 (회귀, 1\~30) | 향후 30일 중 몇째 날에 피크가 오는가 | 보조 타겟 |

두 타겟은 동일한 피처를 사용하며, 모델 아키텍처도 동일하고 출력층의 타겟만 교체한다. XGBoost 기준 15초 × 2 = 30초로 추가 비용이 거의 없다.

### 5.1.1 타겟 ①: virality_score

### 산출 공식

```python
# 특정 키워드의 t시점에서:
baseline_90d = mean(search_trend[t-90 : t-1])    # 과거 90일 평균
std_90d = std(search_trend[t-90 : t-1])           # 과거 90일 표준편차
future_peak = max(search_trend[t+1 : t+30])       # 향후 30일 중 최대값

virality_score = (future_peak - baseline_90d) / (std_90d + ε)    # ε = 1e-6
```

### 학술적 근거

이 공식은 본질적으로 **Z-score (표준 점수)** 의 변형이며, 학술적으로 확립된 이상치/바이럴 탐지 방법론.

**핵심 선행 연구:**

| 논문 | 출처 | Z-score 활용 방식 |
|------|------|------------------|
| "Evaluating the effect of viral posts on social media engagement" (2024) | Nature Scientific Reports | Facebook Z > 3, YouTube Z > 2.5를 바이럴 임계값으로 사용. **본 공식과 동일 구조** |
| "Burst: real-time events burst detection in social text stream" (2021) | Journal of Supercomputing | Z-score로 이벤트 버스트 탐지 |
| "Dive into Time-Series Anomaly Detection: A Decade Review" (2024) | arXiv | Z-score를 시계열 이상치 탐지의 기초 통계 기법으로 분류 |

**분모 smoothing에 대해:**

원래 `std + 1`을 사용했으나, **epsilon smoothing (`std + 1e-6`)으로 수정**. 이유:
- `+1`은 add-one smoothing(Laplace smoothing)으로 실무에서 쓰이긴 하나, 표준편차가 작은 키워드(예: std=0.5)에서 분모가 1.5가 되어 score가 인위적으로 축소됨
- `+ε`(epsilon, 1e-6)은 수치 안정성만 확보하면서 원래 Z-score 값을 보존
- 네이버 DataLab 데이터의 std는 보통 3\~15 범위이므로 실질적 차이는 크지 않으나, 학술적으로 epsilon이 더 정석적

**Z-score의 한계와 보완:**
- Z-score는 **정규분포를 가정**하나, 소셜/검색 데이터는 종종 우측으로 치우친(right-skewed) 분포
- 대안으로 Mahalanobis distance(비정규 다변량), CUSUM(누적합 변화점 탐지)이 있으나, 본 프로젝트에서는 Z-score의 직관성과 구현 단순성이 학부 캡스톤에 더 적합
- 필요 시 `log(virality_score + 1)` 변환으로 분포 치우침 완화 가능

### 산출 예시

| 키워드 | 90일 평균 | 90일 표준편차 | 향후 30일 최대값 | virality_score |
|--------|----------|-------------|-----------------|----------------|
| 약과 | 15 | 5 | 75 | (75-15)/(5+1) = **10.0** |
| 크로플 | 25 | 8 | 60 | (60-25)/(8+1) = **3.9** |
| 초코파이 | 40 | 8 | 48 | (48-40)/(8+1) = **0.9** |
| 현미 크래커 | 10 | 3 | 8 | (8-10)/(3+1) = **-0.5** |

### 분류 변환

```python
viral_label = 1 if virality_score > 2.0 else 0
```

Z-score > 2.0 기준으로 약 80\~120개 바이럴 사례 예상. Z-score > 3.0으로도 실험하여 **임계값에 따른 모델 성능 변화**를 분석 포인트로 활용.

### 90일 윈도우 선택 이유

- 너무 짧으면(7\~30일): 일시적 변동에 민감, 기준선이 불안정
- 너무 길면(180\~365일): 계절성을 지나치게 평활화, 최근 추세 반영 못함
- 90일: 계절성 일부 흡수 + 최근 추세 반영의 적정선
- 기업 모니터링 사례: AppDynamics 등에서 90일을 표준 baseline으로 사용

**주의: 90일이 "정답"이라는 논문은 없음.** 최적 윈도우 크기는 도메인 의존적이며, 학술 연구에서도 "최적 윈도우 탐색은 가장 어려운 과제 중 하나"로 언급 (arXiv:2412.20512). 따라서 **30일, 60일, 90일, 180일로 실험하여 최적값을 탐색**하고, 그 비교 결과 자체를 분석 포인트로 발표에 포함할 것.

### 5.1.2 타겟 ②: peak_timing

#### 산출 공식

```python
# 특정 키워드의 t시점에서:
peak_timing = argmax(search_trend[t+1 : t+30]) + 1    # 1~30 (몇째 날에 피크)
```

향후 30일 시계열에서 최대값이 나타나는 위치(일수)를 타겟으로 사용. `argmax`는 0-indexed이므로 +1하여 1\~30 범위로 변환.

#### 왜 별도 타겟인가

virality_score는 "얼마나 급등하는가"를 알려주지만, "언제 급등하는가"는 알려주지 못한다. 동일한 virality_score = 5.0이라도 3일 후 피크 vs 25일 후 피크는 대응 전략이 완전히 다르다.

시계열 예측(매일의 값을 맞히는 방식)으로도 피크 시점을 알 수 있지만, 오차가 누적되어 30일 후에는 ±25까지 벌어질 수 있다. peak_timing은 "전체 결론(몇째 날인가)"만 맞히면 되므로 훨씬 안정적이다.

#### 산출 예시

| 키워드 | 향후 30일 시계열 (요약) | peak_timing |
|--------|----------------------|-------------|
| 약과 | [23, 28, 45, **75**, 60, 40, ...] | **4일** |
| 크로플 | [25, 26, 27, 30, 35, 40, 48, 52, **60**, 55, ...] | **9일** |
| 초코파이 | [40, 42, **48**, 46, 44, 41, ...] | **3일** |

#### peak_timing 특성

- **범위**: 1\~30 (정수이나 연속값으로 회귀 처리)
- **분포**: 균등하지 않음. 바이럴 키워드는 초반(1\~10일)에 피크가 몰리고, 비바이럴 키워드는 분산됨
- **평가 지표**: MAE가 주 지표 (예: MAE=3.2이면 "평균 3.2일 오차로 피크 시점 예측")
- **분류 변환**: `peak_timing ≤ 7` → "즉시 대응 필요", `8\~14` → "1\~2주 내 대응", `15+` → "여유 있음"

#### 모델 학습 방식

virality_score와 동일한 피처, 동일한 모델 아키텍처를 사용하되 타겟만 교체:

```python

model_virality = XGBRegressor().fit(X_train, y_virality)
model_timing = XGBRegressor().fit(X_train, y_timing)    # 동일 X, 다른 y

# LSTM — 마지막 Dense 출력만 다름
# model 구조 동일, target만 교체하여 별도 학습
```

추가 학습 시간: XGBoost 15초, LSTM 12분. 기존 대비 2배이나 절대 시간이 짧아 부담 없음.

## 5.2 평가 지표 설계

### 회귀 모델 평가 (XGBoost Regressor, LSTM Regressor)

#### 주 지표: RMSE (Root Mean Squared Error)

```
RMSE = √( Σ(predicted_score - actual_score)² / n )
```

- 큰 오차에 높은 페널티 → virality_score 10인 키워드를 2로 예측하는 실수를 집중 포착
- 단독으로는 "이 값이 좋은 건가?" 직관적 판단 어려움 → 보조 지표와 함께 사용

#### 보조 지표 1: MAE (Mean Absolute Error)

```
MAE = Σ|predicted_score - actual_score| / n
```

- "평균적으로 virality_score를 얼마나 틀리는가"
- MAE = 0.8이면 "평균 0.8점 오차" — RMSE보다 직관적
- RMSE와 MAE의 차이가 크면 → 일부 극단적 오예측이 존재한다는 신호

#### 보조 지표 2: Top-K Precision (**발표용 킬러 지표**)

```
실제 virality_score 상위 10개 키워드 vs 모델 예측 상위 10개 키워드
겹치는 개수 / 10 = Top-10 Precision
```

- **발표 시 활용**: "모델이 추천한 바이럴 키워드 10개 중 7개가 실제로 바이럴 되었습니다" (Top-10 Precision = 70%)
- 심사위원에게 가장 직관적이고 임팩트 있는 지표
- Top-5, Top-10, Top-20으로 다양하게 측정 권장

#### 보조 지표 3: Spearman 순위 상관계수

```
ρ = 1 - (6 × Σd²) / (n × (n²-1))
  d = 예측 순위 - 실제 순위
```

- 예측한 순위와 실제 순위의 상관관계 (-1 \~ +1)
- 1에 가까울수록 순위 예측이 정확
- Top-K가 상위만 보는 반면, 이건 **전체 순위 정확도**

### 분류 변환 시 평가 (virality_score > 2.0 기준)

#### 주 지표: F1-Score

```
Precision = TP / (TP + FP)  → "바이럴이라고 한 것 중 실제 바이럴 비율"
Recall    = TP / (TP + FN)  → "실제 바이럴 중 모델이 잡은 비율"
F1        = 2 × (P × R) / (P + R)
```

**왜 Accuracy가 아닌 F1인가**: 바이럴 비율이 10%면, "전부 비바이럴"이라고 답해도 accuracy 90%. 무의미한 지표. F1은 precision과 recall의 균형을 봄.

#### 보조 지표: ROC-AUC

- 임계값(2.0)을 0\~∞로 변화시키면서 True Positive Rate vs False Positive Rate를 그린 곡선의 면적
- 0.5 = 랜덤, 1.0 = 완벽
- "어떤 임계값을 잡든 전반적으로 분류를 잘 하는가"를 종합 평가

### peak_timing 평가

#### 주 지표: MAE (Mean Absolute Error)

```
MAE = Σ|predicted_day - actual_day| / n
```

- MAE = 3.2이면 "평균 3.2일 오차로 피크 시점 예측"
- 실용적으로 ±3일 이내면 충분히 유의미 (주간 단위 의사결정에 활용 가능)

#### 보조 지표: 구간 정확도

```python
# 피크 시점을 3구간으로 나눠서 분류 정확도 측정
bins = {
    "즉시(1~7일)":   peak_timing <= 7,
    "중기(8~14일)":  8 <= peak_timing <= 14,
    "장기(15~30일)": peak_timing >= 15,
}
bin_accuracy = correct_bin_predictions / total
```

- 정확한 날짜를 맞히지 못해도 "어느 구간인가"를 맞히면 실용적 가치 있음
- 발표 시: "피크 시점의 주간 구간을 78% 정확도로 예측" 같은 메시지 가능

### 평가 지표 요약표

| 태스크 | 타겟 | 주 지표 | 보조 지표 | 발표용 핵심 |
|--------|------|---------|-----------|------------|
| 회귀 (강도) | virality_score | RMSE | MAE, Spearman ρ | **Top-10 Precision** |
| 회귀 (시점) | peak_timing | MAE | 구간 정확도 (3-bin) | **"평균 N일 오차로 피크 예측"** |
| 분류 | viral_label | F1-Score | ROC-AUC | P/R 트레이드오프 그래프 |

|------|---------|-------------|-------------|-------------|----------------|----------------|---------------|
| 2023-03-01 | 약과 | 23 | 45 | 12 | 28 | 5 | 12 |
| 2023-03-02 | 약과 | 28 | 52 | 15 | 30 | 8 | 14 |

#### 2-B. 결측치 처리

| 상황 | 처리 방법 |
|------|----------|
| 공휴일에 한쪽 데이터 없음 | Forward fill (전날 값으로 채움) |
| 검색량 너무 낮아서 0 반환 | 0으로 유지 (네이버가 "임계값 미만"을 0으로 줌) |
| 파생 피처 분모가 0 | 분모에 +1 추가 (예: velocity 계산 시) |
| 썸트렌드에 해당 키워드 데이터 없음 | 해당 키워드 제외 또는 0으로 채움 |

#### 2-C. 정규화 관련 주의

- **네이버 DataLab 0\~100 값**: 조회 기간 내 상대값. **반드시 모든 키워드를 같은 기간으로 조회**
- **XGBoost**: 트리 기반이므로 피처 정규화 불필요
- **LSTM**: 모든 입력 피처를 정규화 필수 (Min-Max 또는 StandardScaler)
  - 정규화 기준(mean, std)은 **학습 데이터에서만 계산**하고 테스트 데이터에 동일 적용
  - 테스트 데이터 포함하여 정규화하면 → 데이터 누수(leakage)

### Stage 3: 피처 엔지니어링 + 타겟 생성

#### 3-A. 피처 생성

```python
# 각 (키워드, 날짜) 조합마다 계산
for keyword in keywords:
    for t in date_range:
        # Group A: 검색 모멘텀 (멀티 윈도우: 3d, 7d, 14d, 30d)
        velocity_3d = (trend[t] - trend[t-3]) / (trend[t-3] + 1)
        velocity_7d = (trend[t] - trend[t-7]) / (trend[t-7] + 1)
        velocity_14d = (trend[t] - trend[t-14]) / (trend[t-14] + 1)
        acceleration = velocity_7d[t] - velocity_7d[t-1]
        ma_3, ma_7, ma_14, ma_30 = [mean(trend[t-n+1:t+1]) for n in [3,7,14,30]]
        ma_ratio_3_14 = ma_3 / (ma_14 + 1)
        ma_ratio_7_30 = ma_7 / (ma_30 + 1)
        volatility_3d, volatility_7d, volatility_14d = [std(trend[t-n+1:t+1]) for n in [3,7,14]]
        slope_30d = linear_regression_slope(trend[t-29:t+1])

        # Group B: 소셜 버즈
        mention = blog[t] + news[t]
        mention_velocity = (mention[t] - mention[t-7]) / (mention[t-7] + 1)
        pos_ratio = positive[t] / (total[t] + 1)
        sent_momentum = pos_ratio[t] - pos_ratio[t-7]
        blog_news_ratio = blog[t] / (news[t] + 1)

        # Group C: 교차
        m_s_ratio = mention[t] / (trend[t] + 1)
        lag_corr = correlation(mention[t-7:t], trend[t-4:t+3])

        # Group D: 컨텍스트
        dow = day_of_week(t)
        mon = month(t)
        holiday = is_holiday(t)
        competing = count(other_keywords where virality > 1.5 at time t)

        # Group E: 연관어 벡터 (TF-IDF + PCA, 사전 계산된 20차원 벡터 조회)
        pc_1, pc_2, ..., pc_20 = keyword_vectors[keyword]  # 정적 값

        # Group F: 거시경제 지표 (economic_indicators.csv에서 월 매칭)
        cpi = econ[year_month(t)]['cpi']
        csi = econ[year_month(t)]['csi']
```

#### 3-B. 타겟 생성

```python
baseline_90d = mean(trend[t-90:t-1])
std_90d = std(trend[t-90:t-1])
future_window = trend[t+1:t+30]
future_peak = max(future_window)

# 타겟 ①: 바이럴 강도
virality_score = (future_peak - baseline_90d) / (std_90d + 1e-6)

# 타겟 ②: 피크 시점
peak_timing = argmax(future_window) + 1    # 1~30일
```

**데이터 손실 구간:**
- 처음 90일 (2022.01\~03): baseline 계산에 소요 → 피처 불완전
- 마지막 30일: future_peak 계산 불가 → 타겟 없음
- **실사용 구간: 2022년 4월경 \~ 2025년 8월경**

#### 3-C. LSTM용 시퀀스 생성

```python
# 각 (키워드, 날짜)마다 과거 60일 시계열 추출
for keyword in keywords:
    for t in date_range:
        sequence = raw_features[keyword][t-59 : t+1]  # shape: (60, 8~9)
        target = virality_score[keyword][t]

        # 정규화 (학습 데이터 기준 scaler 적용)
        sequence_normalized = scaler.transform(sequence)

        X_lstm.append(sequence_normalized)
        y_lstm.append(target)
```

### Stage 4: 학습/평가 분할

#### 시간 기준 분할 (랜덤 분할 절대 금지)

```
학습 데이터:  2022.04 ~ 2025.02  (약 35개월)
검증 데이터:  2025.03 ~ 2025.05  (3개월, 하이퍼파라미터 튜닝용)
테스트 데이터: 2025.06 ~ 2025.08  (3개월, 최종 성능 측정용)
```

**왜 랜덤 분할이 금지인가**: 2024년 3월 데이터가 학습에, 2024년 2월 데이터가 테스트에 들어가면 미래 정보로 과거를 예측하는 구조 → 데이터 누수(leakage) → 성능이 비현실적으로 높게 나옴.

#### Rolling Window Validation

학습 데이터를 점점 늘려가면서 평가하는 시계열 전용 교차검증:

```
Fold 1: 학습 2022.04~2023.12 → 평가 2024.01~2024.03
Fold 2: 학습 2022.04~2024.03 → 평가 2024.04~2024.06
Fold 3: 학습 2022.04~2024.06 → 평가 2024.07~2024.09
Fold 4: 학습 2022.04~2024.09 → 평가 2024.10~2024.12
Fold 5: 학습 2022.04~2024.12 → 평가 2025.01~2025.02
```

각 fold의 성능 평균 ± 표준편차를 리포트. 일반적인 k-fold cross-validation은 시계열에서 데이터 누수를 일으키므로 사용 금지.

학습 완료 후, 실제 사용 시:

```
[사용자] 키워드 입력: "약과"

              ↓
[4] 피처 자동 계산 (velocity, acceleration, MA, sentiment 등)
              ↓
[5-A] XGBoost: 피처 43개 벡터 → virality_score + peak_timing 예측
[5-B] LSTM: 60일 시계열 행렬 + 연관어 벡터 → virality_score + peak_timing 예측
              ↓
[6] 결과 출력:
    "약과 — 상승 강도 4.2, 예상 피크 4일 후, 바이럴 가능성 높음 (즉시 대응 필요)"
```

## 5.5 전체 파이프라인 흐름도

```
[Stage 1: 수집]
  쇼핑인사이트 Top500 (분기별) → 키워드 목록 300~500개 확정
  네이버 DataLab API (일별) → search_trend 시계열
  썸트렌드 프리미엄 Excel (일별) → mention, sentiment 시계열
  썸트렌드 연관어 분석 → 키워드별 연관어 30개 수집
  통계청 KOSIS + 한국은행 ECOS → 경제지표 (CPI, CSI)
                    ↓
[Stage 2: 전처리]
  (키워드, 날짜) 기준 Join → 결측치 처리 → 정합성 검증
  연관어 → TF-IDF + PCA → 키워드별 20차원 벡터 생성
  economic_indicators.csv 매핑 → 월별 경제지표 부착
                    ↓
[Stage 3: 피처/타겟 생성]
  XGBoost용: 파생 피처 43개 테이블 (Group A~F)
  LSTM용: 60일 × 10~15채널 시퀀스 + 정규화 + 연관어 벡터 20차원 concat
  타겟: virality_score (연속값) + peak_timing (1~30일) + viral_label (이진)
                    ↓
[Stage 4: 학습/평가 분할]
  시간 기준 분할: 학습 → 검증 → 테스트
  Rolling Window Validation (5 fold)
                    ↓
[Stage 5: 모델 학습]
  Dummy Model + Linear Regression (베이스라인)
  XGBoost Regressor (메인 ①)
  LSTM Regressor (메인 ②)
  Weighted Average Ensemble (XGBoost + LSTM)
                    ↓
[Stage 6: 평가]
  virality_score: RMSE, MAE, Top-K Precision, Spearman ρ
  peak_timing: MAE, 구간 정확도 (3-bin)
  분류 변환: F1, ROC-AUC
  모델 간 비교표 + SHAP 설명
                    ↓
[추론 파이프라인]
  키워드 입력 → API 자동 호출 → 피처 자동 계산 → 모델 예측 → 결과 출력
```
