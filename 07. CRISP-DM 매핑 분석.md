# 07. CRISP-DM 매핑 분석

## 7.1 CRISP-DM이란

CRISP-DM(Cross-Industry Standard Process for Data Mining)은 데이터 마이닝/ML 프로젝트의 표준 프로세스 모델로, 6단계로 구성된다. 각 단계는 순차적이지만, 실제로는 이전 단계로 되돌아가는 반복(iteration)이 빈번하다.

```
[1. Business Understanding] ← → [2. Data Understanding]
            ↓                              ↓
[6. Deployment] ← [5. Evaluation] ← [4. Modeling] ← [3. Data Preparation]
```

## 7.2 현재 문서 매핑 현황

| CRISP-DM 단계 | 커버리지 | 담당 문서 | 상태 |
|---|---|---|---|
| 1. Business Understanding | 60% | 01-프로젝트-개요-및-방향-결정 | 부분적 |
| 2. Data Understanding | 85% | 02-데이터-소스-조사-및-선정 | 잘 됨 |
| 3. Data Preparation | 80% | 03-피처-엔지니어링-설계, 05-파이프라인 | 잘 됨 |
| 4. Modeling | 82% | 04-모델-선택-및-비교-분석 | 잘 됨 |
| 5. Evaluation | 65% | 05-평가-지표, 06-설계-요약 | 부분적 |
| 6. Deployment | 10% | 05 (추론 파이프라인만) | 거의 없음 |

## 7.3 단계별 상세 분석

### Phase 1: Business Understanding (비즈니스 이해) — 60%

**포함된 내용 (01번 문서):**

- 프로젝트 목표: 식품 키워드 바이럴 강도 + 피크 시점 예측
- 프로젝트 제약: 5명 팀, 1학기, 100만원, ML 경험 없음
- 도메인 선정 근거: 과자/베이커리 (계절성 강함, 바이럴 사례 빈번)
- 추론 시나리오: 키워드 입력 → 자동 수집/계산 → 예측 결과 출력
- 초보자 함정 분석: 생존자 편향, 클래스 불균형, 데이터 누수

**빠진 내용:**

- **이해관계자 분석**: "누가 이 모델을 사용하는가"에 대한 구체적 페르소나가 없음. 식품 마케팅팀, 이커머스 MD, 콘텐츠 크리에이터 등 사용자 유형별 요구사항이 다름. 예를 들어 마케팅팀은 "2주 전에 알아야 캠페인 준비 가능"이라는 시간 제약이 있고, MD는 "발주 리드타임 1주"라는 제약이 있음.
- **비즈니스 KPI 연결**: 바이럴 예측이 실제로 어떤 비즈니스 가치를 만드는가에 대한 정량화가 없음. 예: "피크 시점을 ±3일 이내로 예측하면 재고 폐기율을 N% 줄일 수 있다", "바이럴 키워드를 미리 포착하면 마케팅 ROI가 N% 상승" 같은 연결.
- **기존 솔루션 대비 차별성**: 현재 업계에서 바이럴 예측을 어떻게 하고 있는지(수동 모니터링, 소셜 리스닝 도구 등), 본 프로젝트가 그것 대비 어떤 점이 다른지 비교가 없음.
- **리스크 평가**: 예측 실패 시 비용(false positive: 불필요한 재고 확보, false negative: 바이럴 놓침)에 대한 비대칭 분석 없음.

### Phase 2: Data Understanding (데이터 이해) — 85%

**포함된 내용 (02번 문서):**

- 6개 데이터 소스 전수 비교 (네이버, 구글, 카카오, 트위터, YouTube, 썸트렌드)
- 각 소스의 형식, 기간, 비용, 한계 상세 기술
- 한국 시장 점유율 기반 선정 근거 (네이버 64% > 구글 26%)
- 정규화 주의사항, 결측치 전략, Train-Serving Skew 방지 계획
- 데이터 기간 정당화: COVID-19 구조적 단절, EPP ≥ 2.0 기준

**빠진 내용:**

- **실제 EDA 결과**: 아직 데이터를 수집하지 않았으므로 당연하나, 수집 후 추가해야 할 항목들:
  - search_trend 분포 (평균, 표준편차, 왜도, 첨도)
  - 키워드별 시계열 패턴 유형 (꾸준형, 계절형, 스파이크형) 시각화
  - 결측률: 전체 대비 결측 비율, 채널별 차이
  - 이상치: 봇이나 이벤트로 인한 비정상 급등 건수
- **데이터 품질 메트릭**: completeness(완전성), consistency(일관성), timeliness(적시성) 정량화
- **데이터 사전(Data Dictionary)**: 각 필드의 의미, 범위, 단위를 한 테이블로 정리한 것

### Phase 3: Data Preparation (데이터 준비) — 80%

**포함된 내용 (03번, 05번 문서):**

- 4단계 파이프라인 설계 (수집 → 전처리 → 피처/타겟 생성 → 분할)
- 결측치 처리 전략 (Forward fill, 0 유지, 분모 smoothing)
- 정규화 전략: LSTM용 Min-Max/StandardScaler, XGBoost는 불필요
- Join 전략: (키워드, 날짜) 기준 outer join
- 연관어 TF-IDF + PCA 20차원 벡터 자동화 (sklearn 5줄)
- 경제지표 월별→일별 매핑
- XGBoost용 1행 × 43열 스냅샷 vs LSTM용 60행 × 10~15열 시퀀스

**빠진 내용:**

- **아웃라이어 처리 기준**: search_trend에서 봇이나 시스템 오류로 인한 비정상 급등을 어떻게 판별하고 처리하는가. 예: "전일 대비 10배 이상 급등 후 다음 날 원래 수준 복귀"를 제거/보정하는 규칙.
- **로그 변환 적용 시기**: virality_score의 분포가 우측으로 치우칠 때 `log(score + 1)` 변환을 적용하는 조건과 시점이 미결정.
- **데이터 통합 후 sanity check**: join 후 "행 수가 예상(500키워드 × 1,200일 = 60만)과 맞는가", "null이 급증한 키워드는 없는가" 같은 검증 절차.
- **LSTM 정규화 scaler 관리**: 학습 데이터에서 계산한 scaler를 저장하고, 검증/테스트/추론 시 동일 scaler를 적용하는 구체적 코드 흐름.

### Phase 4: Modeling (모델링) — 82%

**포함된 내용 (04번 문서):**

- 4개 모델의 원리 상세 설명 (Dummy, Linear Regression, XGBoost, LSTM) + Weighted Average 앙상블
- 제외 모델 정당화 (Transformer, Prophet, ARIMA, RF)
- 하이퍼파라미터 권장치 (XGBoost: max_depth 5\~6, lr 0.01\~0.1 등)
- LSTM 주의사항: 정규화, dropout, learning rate, early stopping
- 학습 시간 추정 및 클라우드 비용
- Dual target 구조 (virality_score + peak_timing)
- SHAP 설명 가능성

**빠진 내용:**

- **하이퍼파라미터 튜닝 전략**: Grid Search vs Random Search vs Bayesian Optimization 중 선택 이유와 탐색 범위 명시가 없음. 캡스톤 규모에서는 Random Search(n_iter=50)가 적정하다는 판단 근거 필요.
- **모델 선택 의사결정 규칙**: XGBoost와 LSTM 성능이 비슷할 때(예: RMSE 차이 5% 이내) 어떤 걸 최종 모델로 선택하는가. 설명 가능성 우선? 성능 우선? 양쪽 모두 제공?
- **학습 곡선 분석 계획**: training loss vs validation loss 그래프로 과적합/과소적합 진단하는 방법과 대응 전략.
- **베이스라인 모델 상세화**: "항상 평균값을 예측하는 더미 모델"의 RMSE를 먼저 계산하고, 실제 모델이 이 대비 몇 % 개선인지 보고하는 절차.

### Phase 5: Evaluation (평가) — 65%

**포함된 내용 (05번, 06번 문서):**

- 평가 지표 설계: RMSE, MAE, Top-K Precision, Spearman ρ, F1, ROC-AUC
- peak_timing용: MAE, 구간 정확도 (3-bin)
- 시간 기준 분할 (train/validation/test) + Rolling Window Validation
- 데이터 누수 방지 원칙
- 90일 윈도우 크기 탐색 (30/60/90/180일 비교) 계획

**빠진 내용:**

- **성능 해석 기준**: RMSE = 1.5가 "좋은 건가 나쁜 건가"를 판단할 절대 기준이 없음. 더미 모델(항상 평균 예측) 대비 개선율, 또는 "virality_score 1점 오차는 실무적으로 어느 정도 차이인가"라는 해석이 필요.
- **임계값 결정 근거**: virality_score > 2.0을 바이럴 기준으로 잡았으나, 이 값이 비즈니스 관점에서 왜 적절한지(precision-recall 트레이드오프 그래프에서 최적점인지) 근거 부족. 2.0 vs 3.0 비교 실험은 계획되어 있으나 "최종 선택 기준"이 없음.
- **오류 분석(Error Analysis) 계획**: 모델이 크게 틀린 사례를 분석하는 절차가 없음. 예: "어떤 유형의 키워드에서 자주 틀리는가(희귀 키워드? 계절 키워드?)", "과대 예측과 과소 예측 중 어느 쪽이 더 많은가".
- **프로세스 리뷰**: 성능이 기대에 못 미칠 때 다음 단계 우선순위가 없음. 피처 재설계 vs 모델 변경 vs 데이터 추가 수집 중 무엇을 먼저 시도하는가.
- **모델 간 최종 비교표 양식**: 실험 완료 후 작성할 비교표의 구조(모델 × 지표 × fold)가 미리 정의되어 있으면 실험 계획이 명확해짐.

### Phase 6: Deployment (배포) — 10%

**포함된 내용 (05번 문서 §5.4):**

- 추론 파이프라인 설계: 키워드 입력 → API 호출 → 피처 계산 → 모델 예측 → 결과 출력
- 추론 시 데이터 비용 0원 (무료 API + 무료 티어)
- Train-Serving Skew 방지: 학습 때부터 블로그+뉴스만 사용

**빠진 내용:**

- **배포 형태 결정**: 모델을 어떤 형태로 제공할 것인가가 없음. 학부 캡스톤에서 현실적인 선택지:
  - Streamlit 웹앱 (가장 간단, Python만으로 구현)
  - Jupyter Notebook 데모 (발표용)
  - Flask/FastAPI REST API (확장성 있으나 인프라 필요)
- **모델 저장/로드**: 학습된 모델을 어떤 형식으로 저장하고(pickle, joblib, ONNX), 추론 시 어떻게 로드하는가.
- **모니터링 계획**: 배포 후 모델 성능이 시간이 지나면서 저하되는 "모델 드리프트"를 어떻게 감지하는가. 예: "월 1회 최근 데이터로 Top-K Precision을 측정하고, 일정 수준 이하로 떨어지면 재학습".
- **재학습 주기**: 새로운 바이럴 트렌드가 등장하면 기존 모델로는 예측 불가능. 분기별 또는 반기별 재학습 계획.
- **최종 산출물 목록**: 캡스톤 제출물로 무엇을 준비하는가 (보고서, 코드, 모델 파일, 발표 자료, 데모 영상 등).
- **프로젝트 회고**: lessons learned, 예상과 실제의 차이, 개선 방안.

## 7.4 보완 우선순위

학부 캡스톤의 현실적 범위를 고려하여, 투입 대비 효과가 큰 순서로 정렬:

| 우선순위 | 항목 | 소요 시간 | 효과 |
|---|---|---|---|
| 1 | Evaluation: 더미 모델 베이스라인 추가 | 10분 (코드 3줄) | 성능 수치의 의미를 해석 가능하게 만듦 |
| 2 | Evaluation: 모델 비교표 양식 사전 정의 | 15분 | 실험 후 바로 채울 수 있는 틀 확보 |
| 3 | Business: 이해관계자 시나리오 1\~2개 추가 | 20분 | 발표 때 "누구를 위한 모델인가" 설명 가능 |
| 4 | Deployment: 배포 형태 결정 + 모델 저장 방식 | 20분 | CRISP-DM 6단계 완성, 발표 시 완결성 |
| 5 | Deployment: 모니터링/재학습 계획 (계획 수준) | 15분 | 실무 감각 어필 |
| 6 | Modeling: 하이퍼파라미터 튜닝 전략 명시 | 15분 | 실험 계획 구체화 |
| 7 | Data Understanding: EDA 결과 추가 (데이터 수집 후) | 1\~2시간 | 데이터 수집 이후 작업 |

## 7.5 CRISP-DM 반복(Iteration) 계획

CRISP-DM은 일방향이 아니라 반복 프로세스다. 본 프로젝트에서 예상되는 주요 반복 지점:

```
[Modeling → Data Preparation]
  LSTM 성능이 기대 이하 → 채널 구성 변경 (ablation study)
  XGBoost에서 특정 피처 SHAP 기여도 0 → 해당 피처 제거

[Evaluation → Business Understanding]
  Top-K Precision이 50%에 불과 → 임계값(2.0) 재검토
  peak_timing MAE가 10일 → "피크 시점 예측"의 실용 가치 재평가

[Evaluation → Data Preparation]
  특정 키워드 유형에서 오차 집중 → 해당 유형 피처 추가 검토
  계절성 키워드에서 과대 예측 → 계절성 피처 추가 또는 학습 데이터 보강

[Deployment → Evaluation]
  배포 후 실제 성능이 테스트 성능과 다름 → 평가 방법론 재검토
```
